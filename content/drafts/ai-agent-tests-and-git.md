AI 에이전트가 코드를 쓰기 시작한 뒤로, 개발자의 하루 생산량이 몇 배씩 뛰었다는 얘기가 자주 나온다. 과장이 섞여 있긴 하지만, 체감상 틀린 말도 아니다. 한 시간에 할 일을 십 분이면 끝내는 순간이 분명히 있다.

그런데 이렇게 속도가 붙고 나니, 오히려 예전부터 중요하다고 배웠던 두 가지가 더 선명해졌다. 테스트 코드와 git. 둘 다 "기본기"로 묶여서 약간 낡아 보이는 이미지가 있었는데, 에이전트와 같이 일해 보니 이만한 도구가 없다.

## 왜 지금 다시 이 얘기를 하는가

에이전트가 내놓는 코드는 결국 "제안"이다. 그럴듯해 보이지만 돌려보기 전까지는 믿을 수 없다. 내부 변수 이름이 엉뚱하게 바뀌어 있거나, 호출 시그니처가 한 칸 어긋나 있거나, 엣지 케이스가 조용히 무시돼 있거나. 겉으로는 매끈한데 실행하면 터지는 코드를 꽤 자주 만난다.

게다가 에이전트와 일하면 변경 횟수가 폭증한다. 예전엔 한 PR에 커밋 세 개 정도가 일반적이었다면, 에이전트와 한 시간 작업하면 수십 개의 "시도"가 생긴다. 파일 여기저기가 바뀌고, 방향을 잘못 잡았다 싶어서 되돌리고, 다시 붙이고.

이 두 가지, 그러니까 "검증이 필요한 출력"과 "빠르게 쌓이는 변경"을 감당하려면 도구가 필요하다. 테스트 코드가 검증을 맡고, git이 추적과 되돌림을 맡는다. 그래서 이 둘은 에이전트 시대에 덜 중요해진 게 아니라, 훨씬 더 중요해졌다.

## 테스트 코드의 역할

### 가장 빠른 피드백 루프

에이전트가 "네, 구현했습니다"라고 말해도 나는 못 믿는다. 믿을 수 있는 건 초록색 테스트 결과뿐이다. 테스트 한 번 돌리는 게 코드를 한 줄씩 읽는 것보다 훨씬 객관적이고 빠른 검증이다.

특히 로직 변경이 파일 여러 개에 걸쳐 있을 때, 사람이 눈으로 따라가면 금방 피로해진다. 테스트 스위트는 그 피로를 대신 져주는 기계다.

### 회귀 방지

에이전트가 한 영역을 고치면서 다른 영역을 의도치 않게 망가뜨리는 경우가 흔하다. 특히 공통 모듈, 유틸 함수, 설정 파일을 건드릴 때 위험하다. 본인은 한 군데를 "개선"했다고 생각하지만, 실제로는 열 군데가 그 함수를 쓰고 있다.

이럴 때 기존 테스트가 안전망이 된다. "A를 고쳤는데 B가 빨갛게 떴다면 B도 살펴봐야 한다"는 걸 테스트가 자동으로 알려준다. 이건 리뷰만으로는 잡기 어려운 종류의 실수다.

### 요구사항을 코드로 고정하기

에이전트에게 자연어로 스펙을 설명하면 해석이 갈린다. "사용자가 비밀번호를 잊어버렸을 때 이메일로 리셋 링크를 보낸다"라고 적으면, 링크 유효시간, 토큰 포맷, 중복 요청 처리 같은 디테일이 전부 에이전트의 추측으로 채워진다.

테스트 코드는 해석의 여지가 없다. 입력이 이거면 출력이 이거다. 에이전트에게 "이 테스트를 통과시켜"라고 하면, 스펙이 코드로 고정돼서 멋대로 해석할 여지가 줄어든다.

```ts
// 리셋 토큰 생성 동작 명세
test('리셋 토큰은 발급 후 15분간 유효하다', () => {
  const token = createResetToken(user.id, { now: 0 })
  expect(verifyResetToken(token, { now: 14 * 60 * 1000 })).toBe(user.id)
  expect(verifyResetToken(token, { now: 16 * 60 * 1000 })).toBeNull()
})

test('리셋 토큰은 한 번 쓰면 재사용 불가', () => {
  const token = createResetToken(user.id)
  consumeResetToken(token)
  expect(verifyResetToken(token)).toBeNull()
})
```

이런 테스트 두 개만 먼저 쓰고 에이전트에게 "이거 통과하게 구현해줘"라고 하면, "15분"과 "한 번 쓰면 끝"이라는 요구가 코드로 박혀 있어서 엉뚱한 방향으로 가기 어렵다.

### TDD와의 궁합

TDD의 핵심은 "실패하는 테스트부터 쓰고 그걸 통과시키는 최소 구현을 한다"는 것이다. 사람 혼자 할 때는 "빨리 코드부터 쓰고 싶다"는 본능과 싸워야 해서 지키기 어렵다.

그런데 에이전트와 할 때는 이 흐름이 오히려 자연스럽다. 내가 실패하는 테스트를 쓰고, 에이전트에게 구현을 맡긴다. 에이전트는 빠르고, 나는 테스트를 짜는 데 집중한다. 역할이 깔끔하게 갈린다.

무엇보다 에이전트가 "이미 통과하는 테스트를 보고 규칙을 역산"할 수 있기 때문에, 자연어 프롬프트보다 훨씬 정확한 가이드가 된다. TDD가 에이전트 시대에 와서 재평가받고 있는 건 이런 이유다.

## git의 역할

### 원자적 커밋

에이전트와 일할 때 가장 먼저 자리잡아야 하는 습관이 이거다. 한 작업 단위는 한 커밋으로 맞춰둔다. "로그인 엔드포인트 추가"와 "비밀번호 리셋 로직"을 한 커밋에 섞으면, 나중에 한 쪽만 되돌리고 싶을 때 지옥이 된다.

에이전트에게 일을 맡길 때도 범위를 좁힌다. "회원 기능 전체 만들어줘"보다는 "회원가입 핸들러만 추가해줘"가 낫다. 작업이 끝나면 그 단위로 커밋하고 다음으로 넘어간다. 이렇게 하면 커밋 히스토리가 작업 일지처럼 읽힌다.

### 작업 이력 조회

에이전트가 뭘 어떻게 바꿨는지 확인하는 1차 도구는 `git`이다. IDE의 변경 뷰도 괜찮지만, 시간이 지난 변경을 돌아볼 땐 `git log`만 한 게 없다.

```bash
# 최근 커밋 훑기
git log --oneline -20

# 특정 커밋이 뭘 바꿨는지 전체 diff
git show <sha>

# 파일 단위로 누가 뭘 건드렸는지
git log -p -- src/auth/reset.ts
```

에이전트가 만든 코드도 처음엔 리뷰하듯이 `git show`로 한 번씩 본다. 그러다 보면 "이 에이전트는 이런 식으로 코드를 짜는구나" 하는 감이 생기고, 나중엔 어떤 부분을 유심히 봐야 하는지 빨라진다.

### 되돌림이 부담 없어야 한다

에이전트가 방향을 잘못 잡았을 때, "일단 이어서 고쳐보자"는 유혹을 참는 게 좋다. 잘못된 기반 위에 수정을 쌓으면 원본보다 더 이상한 결과가 나온다.

그냥 돌리고 프롬프트를 다시 주는 게 빠르다.

```bash
# 마지막 커밋 날려버리기
git reset --hard HEAD~1

# 커밋 안 한 변경만 초기화
git checkout -- src/auth/reset.ts

# 전체 작업 트리를 직전 상태로
git restore .
```

이게 부담 없이 되려면 결국 커밋이 자주, 의미 단위로 쪼개져 있어야 한다. 커밋이 크면 되돌릴 때 손해 보는 작업도 커진다. 그래서 위의 "원자적 커밋" 습관과 같이 간다.

### rebase로 트리 정리하기

에이전트와 한 시간 일하면 커밋 히스토리가 보통 이렇게 된다.

```
fix typo
fix fix
actually fix this
wip
revert wip
try again
finally works
```

이대로 머지하면 리뷰어가 무슨 일이 있었는지 알 길이 없다. 이럴 때 `git rebase -i`로 묶는다.

```bash
# 최근 커밋 10개를 대화형으로 정리
git rebase -i HEAD~10
```

편집기에서 `pick`을 `squash`로 바꿔 의미 없는 중간 단계를 합치고, `reword`로 커밋 메시지를 다시 쓴다. 결과적으로 "리셋 토큰 로직 추가", "토큰 만료 테스트 추가"처럼 사람이 읽을 수 있는 히스토리만 남긴다.

커밋 메시지는 에이전트가 써도 되지만, 머지 직전에 사람이 한 번 훑는다. 메시지는 "에이전트가 썼다"를 밝히는 게 중요한 게 아니라, "무엇을 왜 바꿨는가"가 담겼느냐가 중요하다. 3개월 뒤에 `git blame`으로 찾아왔을 때 이해가 가야 한다.

### 실험은 브랜치에서

에이전트에게 "이 구조를 완전히 뒤집어서 다시 짜보자"처럼 큰 작업을 맡길 때는 별도 브랜치에서 한다. 결과가 별로면 브랜치를 그냥 버리면 된다. 메인에서 직접 하다가 엉키면 되돌리는 비용이 크다.

```bash
git checkout -b experiment/refactor-auth
# 에이전트에게 리팩토링 맡기기
# 결과가 괜찮으면 머지, 아니면 버림
```

버리는 비용이 낮아야 더 과감한 시도를 해볼 수 있다.

## 실전 워크플로우

### 기능 추가: 테스트 먼저, 에이전트는 나중에

1. 기능 요구를 테스트로 번역한다. 실패하는 테스트를 하나 쓴다.
2. 에이전트에게 "이 테스트를 통과시키는 최소 구현을 해줘"라고 한다.
3. 테스트를 돌린다. 통과하면 커밋.
4. 다음 케이스(엣지 케이스, 에러 처리)에 대한 테스트를 또 쓴다. 2번으로 돌아간다.
5. 기능 완성되면 `git rebase -i`로 커밋을 의미 단위로 묶는다.

```bash
# 반복되는 사이클
npm test -- reset-token.test.ts  # red
# 에이전트에게 구현 요청
npm test -- reset-token.test.ts  # green
git add -p && git commit -m "리셋 토큰 유효기간 15분 적용"
```

한 사이클이 몇 분 안에 끝난다. 속도가 붙으면 에이전트 없이 혼자 하는 것보다 훨씬 빠르다.

### 리팩토링: 그린 상태에서 시작한다

리팩토링의 전제는 "기존 동작이 안 바뀐다"이다. 이걸 보장하는 건 테스트뿐이다.

1. 기존 테스트를 다 돌려서 그린인지 확인한다.
2. 에이전트에게 "이 모듈의 동작은 유지하면서 구조만 이렇게 바꿔줘"라고 한다.
3. 테스트를 다시 돌린다. 여전히 그린이면 커밋. 빨간 게 하나라도 있으면 에이전트가 동작을 바꾼 거다. 커밋하지 말고 질문하거나 롤백한다.
4. 리팩토링 커밋과 기능 변경 커밋을 절대 섞지 않는다.

테스트 없이 리팩토링을 에이전트에게 맡기는 건 도박이다. 겉으로는 깔끔해 보여도 조용히 동작이 바뀐 게 있을 수 있다.

### 버그 픽스: 재현 테스트가 먼저다

1. 버그를 재현하는 테스트를 쓴다. 이 테스트는 반드시 빨갛게 떠야 한다.
2. 에이전트에게 "이 테스트를 초록색으로 만들어줘"라고 한다. 버그 설명은 따로 하지 않아도 테스트가 스펙이다.
3. 테스트 통과를 확인하고, 관련 회귀 테스트도 같이 돌려본다.
4. 한 커밋으로 묶는다. 메시지는 "버그 현상 + 원인" 형태로. 예: "리셋 토큰 만료 시간 UTC 변환 누락 수정"

재현 테스트가 코드베이스에 남으면, 나중에 같은 버그가 재발하는 걸 자동으로 막는다. 버그 픽스 하나당 테스트 하나는 남긴다는 규칙을 지킨다.

## 안 하면 어떻게 되는가

테스트와 git 없이 에이전트와 일하는 걸 한번 본 적 있다. 처음 몇 시간은 엄청나게 빨라 보인다. 에이전트가 쏟아내는 코드를 에디터로 받아서 그대로 돌린다. 그런데 하루가 지나면 코드가 돌아가는지 아닌지 누구도 자신 있게 말을 못 한다. 어디서부터 꼬였는지도 모른다.

에이전트는 빠른 생산 도구다. 빠르게 나오는 결과물에는 빠른 검증과 빠른 되돌림이 붙어야 한다. 테스트가 검증을 맡고, git이 되돌림을 맡는다. 이 둘이 빠지면 그냥 속도만 빠른 혼돈이 된다.

## 정리

에이전트가 주도하는 개발에서 테스트와 git은 생산성의 병목이 아니라 인프라다. 생산된 변경을 사람이 감당 가능한 속도로 이해하고, 롤백하고, 통합하기 위한 장치. 에이전트가 더 잘 써질수록 이 인프라의 필요성은 더 커진다.

둘 다 새 기술도 아니고 화려하지도 않다. 하지만 AI 에이전트를 본격적으로 쓰기 시작한 뒤로 가장 자주 손이 가는 도구가 됐다.
